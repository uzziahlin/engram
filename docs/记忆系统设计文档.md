Version: v2.0
Target Stack: Rust
Target Agents:

* Claude Code

* Aider
* Generic MCP-compatible agents

---

# 1. Project Goal

Build a production-grade long-term memory runtime for coding agents.

The system should:

* Persist engineering knowledge
* Recall debugging history
* Store architectural decisions
* Maintain task timelines
* Retrieve relevant context for future coding sessions
* Continuously consolidate and evolve memory

The system is NOT a chatbot memory system.

The system IS an engineering cognition layer.

# 3. Core Principles

## 3.1 Local-first

All memory stored locally.

No cloud dependency required.

---

## 3.2 Retrieval > Storage

Memory quality depends on retrieval quality.

The main engineering challenge is NOT embedding generation.

The main challenge is:

* routing
* ranking
* compression
* relevance

---

## 3.3 Memory Types Must Be Explicit

Different memory types solve different problems.

Do NOT store everything as vector chunks.

---

## 3.4 Memory Must Be Automatically Generated

The system should infer memory from:

* git events
* task execution
* errors
* terminal logs
* code changes

Avoid requiring users to manually write memories.

---

## 3.5 Transactional Consistency

All writes must complete within a single SQLite transaction.

No async index updates. No dirty flags. No partial writes.

## SQL

```sql
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    entity_type TEXT NOT NULL,
    name TEXT NOT NULL,
    metadata TEXT DEFAULT '{}',

    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,

    UNIQUE(project_id, entity_type, name)
);

CREATE TABLE graph_relations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id TEXT,
    from_entity TEXT NOT NULL,
    to_entity TEXT NOT NULL,
    relation_type TEXT NOT NULL,

    FOREIGN KEY (to_entity) REFERENCES entities(id)
);
```

`graph_relations.project_id` is nullable. `NULL` means a cross-project edge.

# 5.4 Write-Through Pipeline

SQLite is the single source of truth. All writes complete within one transaction.

```text
--- INSERT ---
  SQLite txn begin
    INSERT INTO main memory table
    INSERT INTO FTS5 table (memory_id, ...)
    INSERT/UPDATE entities + graph_relations (if any)
  SQLite txn commit
  Update in-memory Graph (Petgraph)

--- UPDATE ---
  SQLite txn begin
    UPDATE main memory table SET ... WHERE id = ?
    DELETE FROM FTS5 table WHERE memory_id = ?
    INSERT INTO FTS5 table (memory_id, ...) VALUES (...)
    INSERT/UPDATE entities + graph_relations (if any)
  SQLite txn commit
  Update in-memory Graph (Petgraph)

--- DELETE ---
  SQLite txn begin
    DELETE FROM main memory table WHERE id = ?
    DELETE FROM FTS5 table WHERE memory_id = ?
    DELETE FROM graph_relations WHERE from_entity = ? OR to_entity = ?
    (optional) DELETE orphaned entities
  SQLite txn commit
  Update in-memory Graph (Petgraph)
```

FTS5 has no native UPDATE. Use DELETE + INSERT (delete-then-insert).

On transaction failure, all changes roll back. No partial writes.

# 5.5 FTS5 Indexes

Each memory table has a corresponding FTS5 virtual table.

Independent tables (NOT content= external content). Application-layer dual-write within the same transaction.

```sql
CREATE VIRTUAL TABLE episodic_memories_fts USING fts5(
    memory_id TEXT UNINDEXED,
    summary,
    content,
    files_touched,
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE decision_memories_fts USING fts5(
    memory_id TEXT UNINDEXED,
    title,
    context,
    rationale,
    tradeoffs,
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE failure_memories_fts USING fts5(
    memory_id TEXT UNINDEXED,
    incident,
    root_cause,
    fix,
    prevention,
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE procedural_memories_fts USING fts5(
    memory_id TEXT UNINDEXED,
    workflow_name,
    steps,
    related_tools,
    tokenize='porter unicode61'
);
```

`memory_id TEXT UNINDEXED` — stores the main table id for lookup. UNINDEXED prevents UUID tokens from polluting the search index.

## Retrieval Flow

```text
1. SELECT memory_id FROM episodic_memories_fts
   WHERE episodic_memories_fts MATCH ? ORDER BY rank

2. SELECT * FROM episodic_memories
   WHERE id IN (?) AND project_id = ?
```

## Rebuild & Integrity Check

```sql
INSERT INTO episodic_memories_fts(episodic_memories_fts) VALUES('rebuild');
INSERT INTO episodic_memories_fts(episodic_memories_fts) VALUES('integrity-check');
```

Run `rebuild` after data recovery. Run `integrity-check` in test suites.

## Known Limitation: Chinese Tokenization

`unicode61` tokenizer splits Chinese text into individual characters (one token per character). This works but has low precision for Chinese queries.

Mitigation: users can add English keywords via tags.

Phase 2: evaluate `jieba-rs` as a custom FTS5 tokenizer.

# 5.6 Index Strategy

```sql
-- Entities: by project
CREATE INDEX idx_entities_project ON entities(project_id);

-- Graph relations: by project (including cross-project edges)
CREATE INDEX idx_graph_project ON graph_relations(project_id);

-- Graph relations: by entity
CREATE INDEX idx_graph_from ON graph_relations(from_entity);
CREATE INDEX idx_graph_to ON graph_relations(to_entity);

-- Memory tables: by project + time
CREATE INDEX idx_episodic_project_time ON episodic_memories(project_id, created_at DESC);
CREATE INDEX idx_decision_project_time ON decision_memories(project_id, created_at DESC);
CREATE INDEX idx_failure_project_time ON failure_memories(project_id, created_at DESC);
CREATE INDEX idx_procedural_project_time ON procedural_memories(project_id, created_at DESC);
```

# 5.7 Embedding Strategy

MVP: **No embeddings.** Pure BM25 + Graph.

Phase 5+: introduce local embedding model.

* Candidate: `fastembed` (ONNX Runtime) or `candle` (pure Rust)
* Dimension: 384 (MiniLM-L6)
* Storage: `sqlite-vec` extension
* Model update: versioned embeddings, old and new coexist, background recomputation

Do NOT start with embeddings.

Focus first on retrieval quality via BM25 + Graph.

# 6.2 Intent Classification

## Intent Types

```rust
pub enum MemoryIntent {
    Debugging,
    Architecture,
    Workflow,
    Refactor,
    Deployment,
    Incident,
    General,
}
```

## MVP: Rule Engine

Use keyword matching for intent classification.

Define a keyword-to-intent mapping table.

Support compound intents (e.g., Debugging + Incident).

### Fallback Strategy (with Resource Control)

When the rule engine cannot match any intent:

* Query ALL memory sources in parallel
* Reranker merges and ranks results
* Resource limits:
  * Per-source query timeout: 50ms
  * Per-source result cap: 25
  * Reranker input cap: 100
  * Reranker output cap: 20
  * Total fallback latency budget: ≤ 200ms

This sacrifices a small amount of latency for retrieval completeness, avoiding blind spots from intent misclassification.

### Upgrade Path

Rule engine is a temporary solution. Phase 5: upgrade to embedding-based classification.

# 6.4 Hybrid Retrieval

MVP uses SQLite FTS5 BM25 as the sole retrieval engine.

Graph traversal via Petgraph provides supplementary context.

Tantivy is deferred to Phase 2. Introduction conditions:

* FTS5 Chinese tokenization precision insufficient AND custom tokenizer unfeasible
* FTS5 cannot meet custom tokenization needs (e.g., Rust identifier tokenization)
* FTS5 retrieval quality below threshold (Precision@5 < 0.7)

When introduced, Tantivy will use async indexing with dirty flags.

---

# 6.5 Graph Retrieval

Use Petgraph for in-memory graph operations.

## Loading Strategy

```text
Startup:
  SELECT * FROM entities WHERE project_id = ? OR project_id IS NULL
  SELECT * FROM graph_relations WHERE project_id = ? OR project_id IS NULL
  → Build Petgraph in-memory graph

Write:
  Sync update SQLite + in-memory graph within transaction

Cross-project edges:
  project_id IS NULL relations loaded in ALL projects

Scale limit:
  MVP target: ≤ 10,000 nodes
  Startup budget: < 500ms (at 10K nodes)
  > 100K nodes: switch to lazy loading (Phase 2+)
```

## Cycle Detection

Before graph traversal, check `is_cyclic_directed()`.

If cyclic, switch to BFS with visited set.

## Traversal Examples

```text
auth.ts
  -> depends_on
  -> redis.ts

bug_123
  -> fixed_by
  -> commit_456
```

## Phase 2 Optimization Notes

* `OR project_id IS NULL` may bypass index. Fix: UNION ALL two subqueries.
* FTS5 cannot pre-filter by project_id. Fix: add `project_id TEXT UNINDEXED` column to FTS5 tables.

These do not affect MVP performance at ≤ 10K scale.

# 8.4 Token Estimation

MVP uses character-count approximation with content-type weights:

| Content Type | Ratio |
|-------------|-------|
| English text | 1 token ≈ 4 chars |
| Chinese text | 1 token ≈ 2 chars |
| Code | 1 token ≈ 3 chars |

Mixed content calculation with 20% safety margin.

Phase 2+: introduce precise tokenizer (e.g., `tiktoken-rs`).

# 11.3 rmcp Isolation Strategy

`rmcp` is at version 0.x. API may change.

Isolation boundary:

```text
rmcp types → MemoryToolProvider trait → business logic (retrieval/storage)
```

* Define core trait `MemoryToolProvider` to decouple from rmcp concrete types
* MCP server layer depends only on the trait, not rmcp types
* Version lock: `rmcp = "0.2.x"`
* On breaking change: only rewrite trait implementation layer

# 15. Performance Goals

All targets measured under: **10,000 memory records, SQLite WAL mode, FTS5 indexes built, 10,000-node in-memory graph.**

| Operation       | Target  | Conditions |
| --------------- | ------- | ---------- |
| memory insert   | < 20ms  | Including FTS5 index update |
| memory update   | < 25ms  | FTS5 delete-then-insert |
| memory delete   | < 15ms  | Including graph cleanup |
| retrieval       | < 150ms | FTS5 BM25 query |
| graph traversal | < 50ms  | 10K-node in-memory graph |
| graph load      | < 500ms | 10K nodes from SQLite |
| MCP response    | < 300ms | End-to-end including retrieval + context assembly |
