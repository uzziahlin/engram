## 一、检索机制（P0）

> **核心判断**：当前检索链路「看起来比实际智能」——intent/planner 这套机制建了一半就接偏了。
> intent 的唯一实际作用是经 plan 权重间接影响 rerank；而 plan 计算出的「查哪些记忆类型」完全没生效。

### [x] 1.1 P0 · intent→plan 的「记忆源路由」是死代码　✅ 已实现（2026-06-27）

- **现状**：`src/retrieval/planner.rs:36-126` 的 `plan()` 精心计算 `plan.sources`（Debugging→Failure+Episodic、Architecture→Decision+Episodic …），但 `src/mcp/server.rs:631-635` 的 `search_memory` 实际查询完全忽略它——要么 `BM25Retriever::search_all`（`src/retrieval/bm25.rs:130`，查全 4 类），要么按用户指定 `memory_type` 走 `search_by_type`（`src/retrieval/bm25.rs:167`）。grep 确认 `plan.sources` 在生产代码**零消费**，仅 `planner.rs` 自身测试断言引用（`planner.rs:137,138,146,153,160`）。
- **问题**：「基于意图路由到相关记忆源」这一核心卖点未生效。
- **建议**（需决策，二选一）：
  - **(a) 硬过滤**：按 `plan.sources` 只查命中类（复用 `search_by_type` 模式，新增多类型版 `search_by_types`）。General intent 兜底全查、compound intent 取并集——降噪 + 提速。
  - **(b) 软加权**：`search_all` 照旧，rerank 时命中的 source 加分——不改变召回、最安全（复活当年 `graph_weight` 想做未做成的事，见 `docs/code-review-and-fix-plan.md` C1）。
  - **推荐**：默认硬过滤 + 配置开关可退回全查（逃生通道）。
- **涉及文件**：`src/retrieval/bm25.rs`、`src/mcp/server.rs:620-712`、`src/retrieval/planner.rs`
- **成本/风险**：中。风险是硬过滤可能漏跨类结果，靠 General 兜底 + 开关控制。
- **验收**：Debugging 意图查询不再返回 Procedural/Decision；开关关闭时行为与现状一致；新增对应单测。

### [x] 1.2 P0 · rerank 权重硬编码 + compound intent 是「覆盖」非「融合」　✅ 已实现（2026-06-27）

- **现状**：`src/retrieval/planner.rs:38-40` 基础权重 `0.2/0.4/0.4`、`src/retrieval/reranker.rs:36` `W_REL=0.4`、`reranker.rs:50-56` type_prior 全为拍脑袋常量，无任何调优入口。`planner.rs` 多 intent 逐个赋值（last-wins 覆盖）：Debugging 设 `importance=0.6` 后 Incident 又覆盖成 `0.7`，前序 intent 的权重语义被静默丢弃。
- **问题**：无法调优；compound intent 权重退化为「最后一个匹配的 intent」。
- **建议**：
  - 把 4 个全局系数（relevance / recency / importance / type）提为 `[retrieval]` 配置（`src/config.rs` 的 `RetrievalConfig` 已有半结构化基础，`config.rs:141-177`）；
  - planner 内 per-intent 调整改为「从全局基础值缩放」，compound intent 权重改「取 max」（保留最强信号，不丢语义）。
- **涉及文件**：`src/retrieval/planner.rs`、`src/retrieval/reranker.rs`、`src/config.rs`
- **成本/风险**：低。需同步更新现有 reranker 测试（`reranker.rs:93-262`）。
- **验收**：权重可在 `config.toml` 调；compound intent 单测验证取 max；默认值与现状等价。

### [x] 1.3 P0 · intent classifier 用 substring 匹配，误匹配面广　✅ 已实现（2026-06-27）

- **现状**：`src/retrieval/intent_classifier.rs:156` `lower.contains(keyword)`：`fix` 命中 `fixture/suffix`、`merge` 命中 `merge conflict`、中文「迁移」归到 Deployment 但与数据库迁移/代码迁移语义冲突。无词边界、无分词。
- **问题**：意图误判 → 源路由 / 权重偏移。
- **建议**：ASCII 关键词改词边界匹配（手写字符分类判断，不引入 regex 依赖），CJK 保持 `contains`；或弱化为「信号」参与而非强判定。
- **涉及文件**：`src/retrieval/intent_classifier.rs`
- **成本/风险**：低。
- **验收**：`fixture` 不再判为 Debugging；新增边界用例测试。

## 二、智能化机制（P0/P1，差异化所在）

### [x] 2.1 P0 · 检索质量评估闭环缺失（reflection 与调优的前置）　✅ 已实现（2026-06-28）

- **现状**：无任何「这条记忆有没有用」的反馈信号——无采纳 / 引用 / 点击，无命中率、latency、被采纳率指标。除 `RUST_LOG` 外无可观测性。
- **问题**：没有反馈数据，则 1.2 权重调优与 2.2 reflection 都无据可依——这是整个「self-improving retrieval」路线的地基。
- **建议**：工具调用埋点 + 一个轻量 stats 表（query、命中 ids、是否被后续引用）；先做人工可查的度量，**不做**自动学习。
- **涉及文件**：新增 `src/metrics/`（或挂 storage 层）、`src/mcp/server.rs`
- **成本/风险**：中。
- **验收**：能查询某时段 top-k 命中率 / 被采纳率；指标定义写入 `CLAUDE.md` / `ENGRAM.md`。

### [x] 2.2 P1 · reflection engine 是空壳占位　✅ 已实现（2026-06-28）

- **现状**：`src/reflection/engine.rs` 仅 21 行空 struct，README roadmap 第一项未完成，测试覆盖 = 0。
- **问题**：engram 尚未区别于「带搜索的笔记本」。
- **建议**：**先定触发时机与消费者再编码**（否则极易过度设计）。最小闭环：连续 N 次同类 failure → 自动生成一条 procedural / 预防建议，标 tag 待人工确认，确认前不污染主检索。
- **涉及文件**：`src/reflection/engine.rs`（新建逻辑），依赖 2.1 的数据。
- **成本/风险**：高（最易过度设计）。前置依赖 2.1。
- **验收**：能从历史 failure 提炼 ≥1 条可确认规则；人工确认前不进入主检索结果。
- **落地**：新增独立 `reflection_suggestions` 表（pending 建议不在 `procedural_memories` 里，`search_procedural` 天然搜不到，零侵入共享 search 宏）；`ReflectionEngine::reflect` 按 tag 分组（默认阈值 3，`[reflection].min_occurrences` 可配）聚合 `prevention` 为 steps；CLI `reflect`/`suggestions`/`confirm-suggestion`/`reject-suggestion` + MCP 工具 `reflect`/`list_suggestions`/`confirm_suggestion`/`reject_suggestion`；`confirm_suggestion` 复用 `create_procedural` 宏提升为正式 procedural。lib 187 + e2e 4 全绿、clippy `-D warnings` 通过、手动 e2e 闭环验证两条验收。新表走 `CREATE IF NOT EXISTS`，免 `user_version` bump。

## 三、存储与运维（P1/P2，「只进不出」）

### [x] 3.1 P1 · 无空间回收机制（GC / vacuum / wal_checkpoint 全缺）　✅ 已实现（2026-06-28）

- **现状**：代码中确认无 `VACUUM` / `wal_checkpoint` / `incremental_vacuum`。`src/storage/repository.rs` 的 archive 仅置 `archived_at`（软删），FTS 行不清、embedding 向量不删、WAL 文件只增不减。
- **问题**：长期本地使用 `memory.db` + `-wal` 持续膨胀；缺 `engram vacuum` / `engram gc` 运维命令。
- **建议**：新增 CLI `engram gc`（或 `vacuum`）——物理清理已归档超期记忆 + FTS/embedding 残行 + `wal_checkpoint(TRUNCATE)`，支持 dry-run 预览。
- **涉及文件**：`src/storage/repository.rs`、`src/cli/commands.rs`
- **成本/风险**：低。
- **验收**：gc 后 db 文件缩小、FTS 与主表行数一致、search 行为不变、默认 dry-run 安全。

### [x] 3.2 P1 · schema 迁移无版本管理　✅ 已实现（2026-06-28）

- **现状**：无 `PRAGMA user_version`，靠两个硬编码探针函数（`migrate_add_archived_at`、`migrate_fts5_add_tags`）每次启动跑、靠幂等兜底；`CREATE TABLE IF NOT EXISTS` 对老库不补字段。
- **问题**：每加字段手写探针；无法做破坏性迁移（rename / 改类型）；迁移历史不可追溯；`migrate_fts5_add_tags` 在单个事务里 DROP+重建 4 个 FTS 表 + 全量重灌，大库下是长写事务。
- **建议**：引入 `user_version` + 顺序迁移注册表，迁移幂等且可追溯。
- **涉及文件**：`src/storage/repository.rs`
- **成本/风险**：中。
- **验收**：老库启动后 `user_version` 正确递增；新增一个字段无需再手写探针函数。

### [ ] 3.3 P2 · consolidate near-dup 是 O(N²) Jaccard

- **现状**：`src/consolidation/engine.rs:103-117` 双层循环两两比较，代码自认 "MVP stub"（`engine.rs:22-23`）。
- **问题**：大库平方爆炸。
- **建议**：换 MinHash / LSH，或复用已有 embedding 做近似近邻。
- **涉及文件**：`src/consolidation/engine.rs`
- **成本/风险**：中。需先有大数据量才值得做。
- **验收**：1 万条记忆下 consolidate 在秒级完成。

### [x] 3.4 P2 · related_files 全量加载图到内存　✅ 已实现（2026-06-28）

- **现状**：`src/mcp/server.rs` 的 related_files 首次调用把整个 project 的 entities + relations 全量加载进内存图，而 repo 已有高效索引查询 `get_relations_for_entity`（走 `idx_graph_from/to`）未被使用。
- **问题**：大图首次查询延迟 / 内存偏高。
- **建议**：related_files 改走 repo 索引查询，按需取邻接。
- **涉及文件**：`src/mcp/server.rs`、`src/storage/repository.rs`、`src/graph/engine.rs`
- **成本/风险**：中。
- **验收**：related_files 延迟与图规模解耦；内存峰值下降。

## 四、架构与工程质量（P2）

### [ ] 4.1 P2 · server.rs 是 god object（3032 行，6 职责混居）

- **现状**：`src/mcp/server.rs` 同时承担 传输 / 协议 / 工具注册 / 分发 / 全部业务逻辑 / embedding 管理。4 个 `create_*` 几乎完全重复；工具注册需手动同步 4 处（Input struct + trait + 两个 impl + `list_tools` 向量 + match 分发）；业务 SQL 经 `repo.connection()` escape hatch 泄漏到 MCP 层（related_files / timeline / consolidate 直接写裸 SQL）。测试占文件 37%（~1130 行）。
- **建议**：分步拆——(1) 测试外移；(2) embedding 子系统独立模块 `src/mcp/embedding_service.rs`；(3) `create_*` 泛型化（省 ~200 行）；(4) 工具注册单一事实源（宏生成 schema + 分发）；(5) SQL 回收回 repository。
- **涉及文件**：`src/mcp/server.rs`，可能新增 `src/mcp/{transport,embedding_service}.rs`
- **成本/风险**：中高（纯可维护性，无行为变更）。建议拆成多个小 PR。
- **验收**：各文件职责单一；新增一个工具只需改 1 处；全量测试通过。

### [ ] 4.2 P2 · 测试覆盖盲区

- **建议**：随各功能落地补测；reflection / consolidation 优先。
- **涉及文件**：各模块 `#[cfg(test)]`
- **成本/风险**：低。
- **验收**：关键模块覆盖核心路径；`cargo test` 全绿。

## 建议的推进顺序

1. **先做 2.1（埋点）**——它是 1.2 调优和 2.2 reflection 的数据基础，改动小、收益可度量。
2. **再做 1.1–1.3（检索夯实）**——有了埋点，才能验证改动是否真的提升检索质量（否则改了也无法判断好坏）。
3. **并行做 3.1–3.2（存储运维）**——与检索解耦，防 db 膨胀，可独立推进。
4. **2.2 reflection** 待 2.1 数据积累后启动。
5. **P2 工程债**（4.x）穿插或安排专项处理。

> 备注：本路线图与 README 的 Roadmap 段互补——README 列产品功能（HTTP transport / 多 agent 共享），本文聚焦既有机制的修复与夯实。每完成一条，请把 `[ ]` 改为 `[x]` 并补实际落地的 commit 引用。
