### [x] R1 · GC 误删活跃记忆（Critical，无声数据丢失）　✅ 已实现（方向B：gc_purge_one 带守卫级联）
- **证据**：`src/storage/macros.rs:99-128`（`delete_*` 的 SQL 为 `DELETE FROM {table} WHERE id=?1 AND project_id=?2`，**无 `archived_at IS NOT NULL` 守卫**）；`src/storage/repository.rs:1603-1661`（`gc_archived` 先跨连接收集候选 `1617-1629`，再循环 `delete_*` `1634-1641`）。
- **触发**：`engram gc` 是独立 CLI 进程，与常驻 MCP server 共享同一 DB 文件。gc 收集候选后、物理删除前，另一进程对该 id 调 `restore`（置 `archived_at = NULL`），`delete_*` 仍会物理删除该**当前活跃**记忆（连 FTS/graph/embedding）。不依赖 `worker_threads>1`。
- **修复**：两选一（推荐都做）——
  1. `impl_memory_crud!` 宏的 `delete_*` SQL 加 `AND archived_at IS NOT NULL` 守卫（`macros.rs:103`）；
  2. `gc_archived` 全程包进单一事务（候选收集 + 删除同事务，消除 TOCTOU 窗口）。
- **验收**：新增并发测试——gc 候选收集后对某 id restore，再 delete，断言该记忆仍存活；`cargo test` 全绿。
- **责任方**：omp

### [ ] R14 · DoS 加固 + 整数溢出
- **证据**：

  - `server.rs` 所有 `*Input` 的 query/content/tags/files_touched/count/max_commits/limit/days 无上限；
  - `transport.rs:75`（`handler.handle(req)` 无 `catch_unwind`）+ `Cargo.toml:59`（`panic=abort`，未来任何 panic = 全盘崩溃）；
  - `server.rs:516,532`（`input.days * 86400`，`days=i64::MAX` debug panic / release wrap）。
- **修复**：
  1. transport 改带上限的按字节读取（超限返回 `-32700` 并丢弃该行）；

  4. `input.days.saturating_mul(86400)`。
- **验收**：超大请求被拒且不 OOM；注入 panic 不致进程退出；days 极值不溢出。
- **责任方**：omp
